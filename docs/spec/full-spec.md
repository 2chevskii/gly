# Полная спецификация модуля

Данный файл служит для проектирования и разработки полной спецификации модуля.
Краткую спецификацию см. в [[introduction-spec]].

## Назначение

Модуль предоставляет кастомное представление объектов файловой системы PowerShell:

- `System.IO.FileInfo`
- `System.IO.DirectoryInfo`

Основной сценарий - улучшить визуальный вывод команд, которые возвращают эти объекты, включая:

- `Get-ChildItem`
- `Get-Item`
- команды и функции пользователя, возвращающие `FileInfo` / `DirectoryInfo`

Модуль должен менять только формат отображения объектов в PowerShell host. Он не должен менять сами объекты, их типы, свойства, порядок сортировки, поведение провайдеров файловой системы или семантику стандартных командлетов.

## Цели

- Добавить иконки / глифы для файлов и директорий.
- Добавить цветовое оформление имен файлов и директорий.
- Поддержать настраиваемые темы цветов.
- Поддержать настраиваемые наборы глифов.
- Предоставить встроенную тему и встроенный набор глифов.
- Сохранить совместимость с обычным пайплайном PowerShell.
- Сделать поведение предсказуемым для `Get-ChildItem`, `Get-Item` и других команд, возвращающих `FileInfo` / `DirectoryInfo`.
- Позволить пользователю отключить или упростить оформление, если терминал не поддерживает Nerd Fonts или ANSI escape sequences.

## Не цели

- Модуль не является заменой `Get-ChildItem`.
- Модуль не реализует собственный аналог `ls`, `eza` или `tree`.
- Модуль не меняет порядок файлов и директорий.
- Модуль не добавляет файловый кэш метаданных как обязательную часть MVP.
- Модуль не обязан поддерживать все возможности GNU `ls` или `eza`.
- Модуль не должен ломать машинно-читаемые сценарии, где объекты передаются дальше по пайплайну.

## Техническая основа

PowerShell выбирает визуальное представление объектов через format data (`*.format.ps1xml`). Для типов, которые уже имеют встроенные представления, кастомные format data должны иметь более высокий приоритет, чем встроенные.

Для MVP предполагается такая архитектура:

- модуль содержит один или несколько файлов `*.format.ps1xml`;
- при импорте модуль загружает форматные данные через `Update-FormatData -PrependPath`;
- форматные данные определяют кастомные view для `System.IO.FileInfo` и `System.IO.DirectoryInfo`;
- вычисляемые части вывода задаются через `ScriptBlock` / `ExpressionBinding` внутри format view;
- вспомогательные функции форматирования находятся в модуле и вызываются из script block в format data.

Важно: format data в PowerShell действует на всю текущую сессию и не имеет обычной module scope-изоляции. После `Remove-Module` представление может остаться активным до завершения сессии или ручной перезагрузки format data. Это ограничение должно быть явно описано в пользовательской документации.

## Совместимость

Целевые платформы:

- PowerShell 7.x - основная цель.
- Windows PowerShell 5.1 не поддерживается в MVP.

Целевые операционные системы:

- Windows - основная цель.
- Linux / macOS - желательно, если это не усложняет архитектуру MVP.

Терминалы:

- Windows Terminal
- VS Code integrated terminal
- современные ANSI-совместимые терминалы

Шрифты:

- по умолчанию используется набор глифов Nerd Fonts;
- должны существовать альтернативные наборы глифов без Nerd Fonts.

## Основное поведение вывода

При выводе `FileInfo` / `DirectoryInfo` модуль должен отображать:

- индикатор типа объекта;
- глиф / иконку;
- имя файла или директории;
- стандартные файловые метаданные, где это уместно для выбранного view.

Минимальный табличный view по умолчанию для MVP:

| Колонка | Назначение |
| --- | --- |
| Mode | Совместимый с PowerShell индикатор режима / атрибутов |
| LastWriteTime | Время последнего изменения |
| Length | Размер файла; для директорий пустое значение или маркер директории |
| Name | Глиф + цветное имя |

По умолчанию модуль должен влиять только на колонку `Name`, сохраняя привычный layout PowerShell: `Mode`, `LastWriteTime`, `Length`, `Name`.

Human-readable форматирование размера, даты и других метаданных не включается глобально для стандартного `Get-ChildItem` / `Get-Item` table view по умолчанию. Эти настройки применяются в `Show-Gly`, `Show-GlyTree`, `Show-GlyGrid` и дополнительных view / renderer-командах.

Уже в MVP модуль должен также предоставить дополнительные варианты отображения:

- tree view;
- grid view;
- compact view;
- wide view;
- long / detailed view;
- human-readable view для размера, даты изменения и других метаданных.

Архитектурное правило: стандартное форматирование `FileInfo` / `DirectoryInfo` реализуется через `*.format.ps1xml`, а сложные представления вроде дерева и сетки реализуются отдельными renderer-командами модуля. Не нужно пытаться реализовать дерево только через `format.ps1xml`, потому что tree layout требует контроля над обходом директорий, вложенностью и отступами.

Идеи для будущих версий:

- Git-aware view;
- permission-aware view;
- выделение executable-файлов.

## Правила оформления

### Директории

Директории должны иметь отдельный цвет и отдельный глиф.

Предварительное правило:

- обычная директория - базовый directory glyph;
- скрытая директория - отдельный стиль или приглушенный цвет;
- символическая ссылка на директорию - отдельный glyph или link indicator.

### Файлы

Файлы должны оформляться на основе набора правил.

Предварительный порядок выбора правила:

1. Специальное имя файла, например `.gitignore`, `Dockerfile`, `Makefile`.
2. Полное расширение для compound extension, например `.d.ts`, `.tar.gz`.
3. Обычное расширение, например `.ps1`, `.cs`, `.json`.
4. Атрибуты файла, например hidden, readonly, executable.
5. Fallback-правило для обычного файла.

Если несколько правил применимы одновременно, более специфичное правило должно побеждать менее специфичное.

### Символические ссылки и reparse points

Модуль должен различать обычные файлы / директории и ссылки, если информация доступна через `FileSystemInfo.Attributes`.

PowerShell 7 в стандартном выводе показывает target для junction / symlink в колонке `Name` в виде `name -> target`, если target доступен. Поэтому MVP должен сохранить это поведение в стандартном view: кастомное оформление имени не должно скрывать target ссылки.

Расширенное оформление target ссылки, например отдельный цвет target или отдельная колонка, остается идеей для будущих версий.

### Скрытые элементы

Скрытые файлы и директории должны иметь отдельный стиль. На Windows основой является атрибут `Hidden`; на Unix-подобных системах дополнительно может учитываться имя, начинающееся с `.`.

### Исполняемые файлы

Выделение исполняемых файлов по Unix execute bit и Windows-расширениям (`.exe`, `.cmd`, `.bat`, `.ps1`) не входит в MVP, но сохраняется как идея для будущих версий.

## Темы

Тема определяет цвета и стили, но не определяет сами глифы.

Тема состоит из упорядоченного списка правил. Каждое правило содержит selector и описание стиля.

Политика приоритета правил: ниже расположенное правило имеет более высокий приоритет. Это работает как CSS cascade: сначала задаются широкие правила, затем более конкретные overrides.

Предварительная структура темы:

```powershell
@{
    Name = 'Default'
    Default = @{
        Foreground = '#d4d4d4'
        Background = $null
        Bold = $false
        Underline = $false
    }
    Rules = @(
        @{
            Selector = @{
                Kind = 'Directory'
            }
            Style = @{
                Foreground = '#8ec07c'
                Bold = $true
            }
        }
        @{
            Selector = @{
                Glob = '**/*.ps1'
            }
            Style = @{
                Foreground = '#83a598'
            }
        }
        @{
            Selector = @{
                Kind = 'File'
                Attributes = @('Hidden')
            }
            Style = @{
                Foreground = '#928374'
            }
        }
    )
}
```

Предпочтительный вариант для MVP: объектная модель стиля, которая на этапе рендера преобразуется в ANSI SGR-последовательность или в пустую строку, если цвет отключен.

Стиль задается объектом:

```powershell
@{
    Foreground = '#8ec07c'
    Background = $null
    Bold = $false
    Italic = $false
    Underline = $false
}
```

Предварительные поля selector:

- `Glob` - glob-паттерн пути или имени, например `**/*.ps1`;
- `Kind` - тип объекта: `File`, `Directory`, `Symlink`, `Junction`, `HardLink`, `Other`;
- `Attributes` - атрибуты `FileAttributes`, например `Hidden`, `ReadOnly`, `Archive`, `ReparsePoint`;
- `Extension` - расширение или compound extension, например `.ps1`, `.d.ts`, `.tar.gz`;
- `Name` - точное имя файла, например `Dockerfile`, `.gitignore`;
- `UnixMode` - будущий selector для Unix file permissions.

Если `Default` не настроен и объект не подходит ни под одно правило, модуль не применяет стиль: используется стандартный цвет терминала.

Поддерживаемые варианты рендеринга цвета:

- через `$PSStyle`, если версия PowerShell 7.2 или новее;
- через прямые ANSI escape sequences, если версия PowerShell ниже 7.2 или если пользователь явно выбрал этот backend.

PowerShell 7 предоставляет `$PSStyle.FileInfo` со стилями `Directory`, `Executable`, `SymbolicLink` и словарем `Extension`. Модуль должен учитывать этот механизм как часть интеграции с PowerShell, но не обязан использовать его как единственное хранилище темы. Предварительное решение: темы модуля хранятся в собственной объектной модели, а рендеринг использует `$PSStyle` для генерации ANSI-последовательностей и уважает `$PSStyle.OutputRendering`.

Встроенные темы не должны изменяться пользователем напрямую. Для изменения built-in темы пользователь должен скопировать ее в новую пользовательскую тему и зарегистрировать копию.

Модуль должен предоставлять предустановленные темы для разных предпочтений пользователя, включая dark / light варианты. Автоматически определять dark / light фон терминала модуль не должен; выбор темы остается на пользователе.

Список кандидатов для built-in тем ведется в [[theme-candidates]]. Конкретный набор built-in тем MVP выбирается отдельно из этого списка.

## Наборы глифов

Набор глифов определяет символы, которые добавляются к имени объекта.

Набор глифов использует ту же selector-модель, что и тема, но вместо `Style` правило возвращает glyph.

Предварительная структура набора:

```powershell
@{
    Name = 'NerdFonts'
    Default = '...'
    Rules = @(
        @{
            Selector = @{
                Kind = 'Directory'
            }
            Glyph = '...'
        }
        @{
            Selector = @{
                Glob = '**/*.ps1'
            }
            Glyph = '...'
        }
        @{
            Selector = @{
                Name = 'Dockerfile'
            }
            Glyph = '...'
        }
    )
}
```

Если `Default` не настроен и объект не подходит ни под одно правило, модуль не добавляет иконку.

Встроенные наборы глифов MVP:

- `NerdFonts` - набор по умолчанию;
- `ANSI` - читаемые ASCII/ANSI-маркеры;
- `ANSICompact` - компактные ASCII/ANSI-маркеры;
- `Unicode` - Unicode-символы без зависимости от Nerd Fonts;
- `Emoji` - emoji-иконки.

Автоматическое надежное определение наличия Nerd Font не входит в MVP. Шрифт выбирается и рендерится терминальным эмулятором, а PowerShell обычно не имеет надежного API для проверки конкретного активного font face и поддержки конкретных glyph code points. Поэтому выбор glyph set должен быть явной настройкой пользователя.

Модуль не содержит автоматического fallback-механизма между наборами глифов. Если пользователь выбрал `NerdFonts`, модуль использует `NerdFonts`; если glyph не отображается корректно, пользователь переключает `GlyphSet` вручную.

## Конфигурация

Модуль должен поддерживать глобальную конфигурацию текущей PowerShell-сессии.

Предварительные параметры конфигурации:

```powershell
@{
    Enabled = $true
    Theme = 'Default'
    GlyphSet = 'NerdFonts'
    ShowGlyphs = $true
    ShowColors = $true
    SizeFormat = 'Raw'
    DateFormat = 'Default'
    StyleRenderer = 'Auto'
    RespectNoColor = $true
    ResetAfterName = $true
}
```

`StyleRenderer` задает backend рендеринга стилей:

- `Auto` - `$PSStyle` для PowerShell 7.2+, ANSI для более ранних версий;
- `PSStyle` - принудительно использовать `$PSStyle`;
- `Ansi` - принудительно генерировать ANSI escape sequences напрямую;
- `PlainText` - не генерировать управляющие последовательности цвета.

Дополнительно модуль должен поддерживать пользовательскую preference-переменную `$GlyStyleRenderer`. Если она задана в сессии, она имеет приоритет над значением `StyleRenderer` из конфигурации. Это нужно как аварийный ручной override на случай проблем с `$PSStyle` или прямым ANSI-рендерингом.

Конфигурация MVP хранится в памяти текущей сессии. Постоянная пользовательская конфигурация на диске не входит в MVP.

Пользовательские темы и наборы глифов в MVP задаются только как PowerShell `hashtable` / `pscustomobject`, передаваемые в соответствующие cmdlet регистрации. Форматы файлов (`.psd1`, `.json`, `.yaml`, `.ps1`) не проектируются до появления требования на постоянное хранение конфигурации.

## Cmdlets

Публичное имя модуля: `gly`.

Для публичных cmdlet используется noun `Gly`. Это короткий noun, соответствующий имени модуля и не привязанный только к `ls`.

### Enable-Gly

Включает кастомное форматирование в текущей сессии.

Предварительное поведение:

- загружает format data через `Update-FormatData -PrependPath`;
- включает глобальный флаг `Enabled`;
- применяет активную тему и набор глифов.

Модуль должен автоматически активироваться при `Import-Module`. `Enable-Gly` остается публичной командой для повторного включения после `Disable-Gly` или ручного восстановления состояния.

### Disable-Gly

Отключает применение стилей на уровне функций модуля.

Ограничение: PowerShell не предоставляет простой scoped-механизм удаления уже загруженных format data. Поэтому команда должна отключать цвет / глифы через конфигурацию, но не гарантировать полное восстановление встроенного view до новой сессии.

### Get-GlyConfiguration

Возвращает текущую конфигурацию.

### Set-GlyConfiguration

Изменяет параметры текущей конфигурации.

Предварительные параметры:

- `-Theme`
- `-GlyphSet`
- `-ShowGlyphs`
- `-ShowColors`
- `-StyleRenderer`
- `-RespectNoColor`

### Get-GlyTheme

Возвращает доступные темы или конкретную тему по имени.

### Set-GlyTheme

Выбирает активную тему.

### Register-GlyTheme

Регистрирует пользовательскую тему в текущей сессии.

Требования:

- не позволяет перезаписать built-in тему без явного создания темы с новым именем;
- валидирует структуру темы;
- поддерживает регистрацию темы из объекта и, возможно, из файла;
- возвращает зарегистрированную тему.

### Copy-GlyTheme

Создает пользовательскую копию существующей темы, включая built-in тему.

Предназначена для сценария: скопировать предустановленную тему, изменить поля, зарегистрировать под новым именем.

### Get-GlyGlyphSet

Возвращает доступные наборы глифов или конкретный набор по имени.

### Set-GlyGlyphSet

Выбирает активный набор глифов.

### Register-GlyGlyphSet

Регистрирует пользовательский набор глифов в текущей сессии.

Требования:

- не позволяет перезаписать built-in набор без явного создания набора с новым именем;
- валидирует структуру набора;
- поддерживает регистрацию набора из объекта и, возможно, из файла.

### Copy-GlyGlyphSet

Создает пользовательскую копию существующего набора глифов, включая built-in набор.

### Show-Gly

Основная точка входа для интерактивного форматирования объектов файловой системы.

Alias: `gly`.

Команда должна принимать:

- `System.IO.FileInfo` из pipeline;
- `System.IO.DirectoryInfo` из pipeline;
- списки `FileInfo` / `DirectoryInfo`;
- `-Path`;
- `-LiteralPath`.

Поведение:

- применяет текущую тему;
- применяет текущий набор глифов;
- применяет текущий renderer backend;
- использует настройки human-readable размера и даты;
- не изменяет входные объекты;
- возвращает форматированный вывод, предназначенный для интерактивного отображения.

`Show-Gly` не заменяет `Get-ChildItem`. При передаче `-Path` / `-LiteralPath` команда может внутри использовать `Get-ChildItem` / `Get-Item`, но должна явно документировать это поведение.

### Show-GlyTree

Renderer-команда для древовидного вывода файловой системы.

Команда должна управлять обходом директорий, глубиной, отступами и отображением промежуточных элементов. Она не должна заменять `Get-ChildItem`, но может принимать путь и/или объекты из пайплайна.

Alias: `glytr`.

### Show-GlyGrid

Renderer-команда для сеточного вывода.

Команда должна использовать активные тему и набор глифов, но самостоятельно управлять layout сетки.

Alias: `glygr`.

Дополнительные aliases для выбора тем и наборов глифов в MVP не добавляются.

Предварительные переменные модуля:

| Переменная | Назначение |

| `$script:GlyConfiguration` | Текущая конфигурация сессии |
| `$script:GlyThemes` | Реестр доступных тем |
| `$script:GlyGlyphSets` | Реестр доступных наборов глифов |
| `$script:GlyFormatDataPath` | Путь к активному `*.format.ps1xml` |

Публичную переменную полной конфигурации в MVP не предоставляем. Внешнее управление конфигурацией выполняется через cmdlet API. Исключение: `$GlyStyleRenderer`, так как это диагностический и compatibility override.

### Требования к внутренним функциям

- Не должны изменять входной объект.
- Не должны выбрасывать необработанные исключения во время форматирования.
- При ошибке должны возвращать безопасный fallback.
- Должны быть достаточно быстрыми для директорий с большим количеством элементов.

## Производительность

Форматирование вызывается для каждого объекта в выводе, поэтому критичны:

- минимальное количество filesystem-запросов;
- отсутствие чтения содержимого файлов;
- отсутствие Git-запросов в MVP;
- предварительно подготовленные lookup-таблицы для расширений и имен;
- отсутствие дорогих операций в script block format data.

Предварительное ограничение MVP: оформление должно использовать только данные, уже доступные на объекте `FileInfo` / `DirectoryInfo`, плюс статические таблицы правил.

Git-статусы не входят в MVP из-за высокой стоимости и сложности корректного определения статуса для каждого элемента. Идея сохраняется для будущей Git-aware версии.

## Ошибки и fallback

Если тема, набор глифов или конкретное правило не найдены, модуль должен использовать безопасное поведение:

- стандартный цвет терминала, если стиль не найден;
- отсутствие glyph, если glyph не найден;
- отсутствие цвета, если ANSI недоступен;
- обычное имя объекта, если форматирование полностью отключено.

Модуль не должен ломать вывод `Get-ChildItem`, если конфигурация повреждена.

## Учет стандартов терминала

Модуль должен учитывать:

- `$PSStyle.OutputRendering`, если доступно;
- переменную окружения `NO_COLOR`, если включен `RespectNoColor`;
- отсутствие интерактивного host, если вывод перенаправлен;
- необходимость сбрасывать ANSI-стиль после имени.

Модуль не должен пытаться автоматически определить:

- установлен ли у пользователя Nerd Font;
- светлая или темная тема терминала;
- точные возможности конкретного terminal emulator сверх доступных сигналов PowerShell и переменных окружения.

Эти параметры задаются пользователем через конфигурацию.

## Тестирование

Минимальный набор тестов:

- выбор glyph по точному имени;
- выбор glyph по compound extension;
- выбор glyph по обычному extension;
- fallback для неизвестного файла;
- отдельное оформление директорий;
- отдельное оформление hidden-файлов;
- отключение glyph;
- отключение цвета;
- поведение при неизвестной теме;
- поведение при неизвестном наборе глифов.

Интеграционные проверки:

- импорт модуля не падает;
- `Get-ChildItem` выводит кастомный view;
- `Get-Item` выводит кастомный view;
- объекты в пайплайне остаются `FileInfo` / `DirectoryInfo`;
- `Format-Table`, `Format-List`, `Select-Object` продолжают работать ожидаемо.

## Документация

Документация должна явно описывать:

- требования к Nerd Fonts;
- альтернативные наборы глифов без Nerd Fonts;
- особенности ANSI-цветов;
- ограничение PowerShell format data: форматирование действует на всю сессию;
- ограничение `Disable-Gly`;
- примеры включения в `$PROFILE`;
- примеры выбора темы и набора глифов.
